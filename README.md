# confluence-label-bot

Бот-демон для Confluence **Server / Data Center**. Периодически находит страницы
с заданными лейблами внутри поддеревьев страниц-источников и переносит их под
страницу-назначение. Связок «откуда → куда по каким лейблам» может быть несколько —
они описываются правилами в `rules.yaml`.

## Как работает

1. Для каждого правила обходит **всё поддерево** источника на любой глубине,
   рекурсивно спускаясь по `GET /rest/api/content/{id}/child/page`
   (`expand=version,space,metadata.labels`), и отбирает страницы, у которых есть
   любой из указанных лейблов. Регистр лейбла не важен.

   Обход намеренно сделан вместо поиска по CQL: `/content/search` читает из
   Lucene-индекса, который обновляется асинхронно, а в кластере на каждой ноде
   свой — из-за чего CQL умеет молча возвращать неполную выдачу. Обход дерева
   читает из базы и от состояния индекса не зависит.

   Цена — запрос на каждый узел поддерева вместо одного на всё правило. Путь
   предков бот строит по ходу спуска, поэтому `expand=ancestors` не запрашивает
   (на `/descendant/page` он приводил к HTTP 500).
2. Каждую найденную страницу переносит под `target` правила — сменой родителя
   (`PUT /rest/api/content/{id}` с новым `ancestors` и версией +1). Дочерние
   страницы переносятся вместе с ней.
3. Повторяет цикл по расписанию `CRON_SCHEDULE`.

Правила применяются по порядку; ошибка одного правила не останавливает остальные.

Операция идемпотентна: после переноса страница выходит из поддерева источника и
на следующих проходах уже не выбирается.

## Настройка

```bash
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/macOS:
source .venv/bin/activate

pip install -r requirements.txt

cp .env.example .env               # подключение и авторизация
cp rules.example.yaml rules.yaml   # правила переноса
```

### Правила переноса (`rules.yaml`)

Каждое правило — связка «откуда → куда по каким лейблам»:

```yaml
rules:
  - name: drafts-to-published  # необязательно, для логов
    source: "111111"           # ID страницы-источника: берётся всё её поддерево
    labels: [ready, approved]  # лейблы-триггеры: достаточно любого (ИЛИ)
    target: "333333"           # ID страницы-назначения
    space: DOCS                # необязательно: доп. фильтр по пространству

  - name: archive              # один лейбл можно писать без списка
    source: "222222"
    labels: to-archive
    target: "444444"
```

| Поле | Несколько значений | Смысл |
|---|---|---|
| `source` | нет, ровно один | Откуда: страницы из поддерева этого ID |
| `labels` | да, по ИЛИ | По каким лейблам отбирать |
| `target` | нет, ровно один | Куда переносить |
| `space` | нет, необязательно | Ограничить отбор пространством |
| `name` | нет, необязательно | Имя правила в логах |

`source` и `target` — всегда по одной странице. Для `target` иначе и нельзя:
в Confluence страница не может лежать под несколькими родителями. `source`
дополнительно проверяется на уникальность **между правилами** — одну
страницу-источник обслуживает ровно одно правило, иначе непонятно, куда
переносить подходящую под оба правила страницу.

Файл валидируется при старте: пустые поля, списки в `source`/`target`, `target`,
совпадающий с `source`, повторяющиеся имена правил или источники и опечатки в
названиях полей приводят к понятной ошибке до обращения к API.

### Переменные `.env`

| Переменная | Назначение |
|---|---|
| `CONFLUENCE_BASE_URL` | URL инсталляции, без завершающего `/` |
| `CONFLUENCE_PAT` | Personal Access Token (рекомендуется) |
| `CONFLUENCE_USERNAME` / `CONFLUENCE_PASSWORD` | Basic-авторизация, если нет PAT |
| `CONFLUENCE_VERIFY_SSL` | Проверять SSL (`false` для самоподписанных сертификатов) |
| `CONFLUENCE_CA_CERT_DIR` | Папка с корневыми (Root/CA) сертификатами для доверенной проверки TLS |
| `RULES_FILE` | Путь к файлу правил (по умолчанию `rules.yaml`) |
| `CRON_SCHEDULE` | Расписание запусков, cron-выражение (по умолчанию `* * * * *`) |
| `DRY_RUN` | `true` — только логировать, ничего не менять |
| `CONFLUENCE_QUERY_DELAY` | **Обязательно.** Минимальный интервал (сек) между запросами к Confluence — защита от HTTP 429. `0` — без паузы |
| `CONFLUENCE_MAX_RETRIES` | Сколько раз повторить запрос при 429 (по умолчанию `3`) |
| `CONFLUENCE_RETRY_MAX_WAIT` | Верхняя граница (сек) одиночного ожидания перед повтором (по умолчанию `60`) |
| `HEALTH_PORT` | Порт health-сервера в режиме демона (по умолчанию `8080`) |
| `HEALTH_LIVENESS_TIMEOUT` | Порог liveness (сек): дольше этого без признаков жизни — под перезапускается (по умолчанию `300`) |

> ID страницы виден в URL (`…/pages/viewpage.action?pageId=12345`) либо в
> «Page Information» → «Page ID».

### Расписание

`CRON_SCHEDULE` — обычное cron-выражение «мин час день месяц день_недели» в
локальном времени машины, где работает бот:

| Выражение | Когда |
|---|---|
| `* * * * *` | каждую минуту |
| `*/5 * * * *` | каждые 5 минут |
| `0 * * * *` | в начале каждого часа |
| `0 9 * * 1-5` | в 9:00 по будням |
| `0 9,18 * * *` | в 9:00 и в 18:00 |

Выражение проверяется при старте, при опечатке бот не запустится. Перед каждым
ожиданием он пишет в лог время следующего запуска. Срок отсчитывается от момента
окончания прохода, так что если проход затянулся дольше промежутка между
запусками, пропущенные сроки не копятся — бот просто идёт к ближайшему будущему.

Планировать можно и снаружи — systemd timer или «Планировщик заданий» Windows,
запуская `python -m confluence_label_bot --once`.

### Троттлинг и защита от 429

Бот обходит поддерево по **одному запросу на страницу**, поэтому на больших
деревьях легко упереться в лимит Confluence и получить `HTTP 429 Too Many
Requests`. Обязательный `CONFLUENCE_QUERY_DELAY` задаёт минимальный интервал
между любыми двумя запросами: пауза добавляется, только если запросы идут чаще
(при редких запросах не тормозит). Значения `0.1–0.5` обычно достаточно; `0`
отключает паузу.

Если 429 всё же прилетел, бот повторяет запрос до `CONFLUENCE_MAX_RETRIES` раз,
уважая заголовок `Retry-After` (а без него — экспоненциальный backoff). Разовое
ожидание ограничено сверху `CONFLUENCE_RETRY_MAX_WAIT`, чтобы огромный
`Retry-After` не подвесил бота надолго.

### Health-check (Kubernetes)

В режиме демона бот поднимает лёгкий HTTP-сервер здоровья (stdlib, без внешних
зависимостей) на `0.0.0.0:HEALTH_PORT`:

| Эндпоинт | Проба | 200, когда |
|---|---|---|
| `GET /healthz` | liveness | главный цикл жив и не завис |
| `GET /readyz` | readiness | прошла стартовая проверка связи с Confluence |

Иначе — `503` с JSON-телом (у `/healthz` — возраст heartbeat, у `/readyz` —
результат последнего прохода).

**Важно про порог liveness.** Heartbeat означает «главный поток отвечает», а не
«давно ли был прогон»: бот обновляет его и во время работы, и во время сна между
запусками по cron. Поэтому `HEALTH_LIVENESS_TIMEOUT` **не зависит** от
`CRON_SCHEDULE` — это «максимум, сколько главный поток может молчать». Ставьте
его с запасом на самый долгий одиночный запрос (timeout 30с + ретраи 429), а не
на интервал между прогонами. Будь порог привязан к интервалу, при редком
расписании (напр. `0 6 * * *`) перезапуск днём привёл бы к CrashLoopBackOff:
живой под спал бы до утра, а проба убивала бы его каждые несколько минут.

Пример проб в манифесте Deployment:

```yaml
        ports:
          - name: health
            containerPort: 8080
        livenessProbe:
          httpGet: { path: /healthz, port: health }
          periodSeconds: 30
          failureThreshold: 3
        readinessProbe:
          httpGet: { path: /readyz, port: health }
          periodSeconds: 30
        startupProbe:            # даём время на стартовую проверку связи
          httpGet: { path: /readyz, port: health }
          failureThreshold: 30
          periodSeconds: 5
```

Разовый (`--once`) и проверочный (`--check`) прогоны сервер не поднимают —
он нужен только долгоживущему демону.

### Корневые сертификаты

Если Confluence подписан внутренним корневым CA, положите его сертификаты
(`.pem` / `.crt` / `.cer` / `.der`) в папку (по умолчанию `certs/`, уже в
`.gitignore`) и укажите её в `CONFLUENCE_CA_CERT_DIR`. При старте все сертификаты
из папки собираются в единый бандл (вместе с `certifi`) и используются для
проверки TLS — отключать `CONFLUENCE_VERIFY_SSL` не нужно.

## Запуск

Активируйте venv (`.venv\Scripts\activate` / `source .venv/bin/activate`), затем:

```bash
python -m confluence_label_bot --check          # проверить правила и доступ к страницам
python -m confluence_label_bot --once           # один проход и выйти
python -m confluence_label_bot                  # демон (по интервалу)
python -m confluence_label_bot --rules my.yaml  # другой файл правил
```

`--check` проходит по всем правилам и показывает заголовки страниц-источников и
назначений — так сразу видно опечатку в ID.

Начните с `DRY_RUN=true` и `--once`, чтобы увидеть, какие страницы будут перенесены,
без внесения изменений.

## Запуск как службы (systemd, пример)

```ini
[Unit]
Description=Confluence label mover bot
After=network-online.target

[Service]
WorkingDirectory=/opt/confluence-label-bot
ExecStart=/opt/confluence-label-bot/.venv/bin/python -m confluence_label_bot
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```
